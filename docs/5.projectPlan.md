# 3. Module-wise Plan

---

## 3.1 Mizhi — Detection & Monitoring (Core Engine)

### Key Decisions

* EfficientNet remains in **PyTorch (.pth)**
  → Required for **Grad-CAM compatibility**

* Thanal remains in **ONNX**
  → Efficient for inference

---

### A. Disease Detection

* Model: EfficientNet-B0 (.pth)
* Classes: 27 diseases + 7 healthy
* Framework: PyTorch

#### Pipeline

```
Input Image
→ Preprocessing
→ EfficientNet (.pth)
→ Prediction + Confidence
→ Grad-CAM heatmap
```

#### Logic

* Confidence threshold: **0.85**
* Below threshold → `UNRELIABLE`
* Above threshold → proceed to advisory

---

### B. VNIR Monitoring (Thanal)

* Model: U-Net Attention (.onnx)
* Preprocessing: HSV leaf isolation

#### Pipeline

```
Image
→ HSV Leaf Mask
→ Thanal (ONNX)
→ NIR Prediction
→ Compute NIR/Green Ratio
```

---

### Monitoring Strategy (Updated)

* No continuous monitoring
* **3 notifications per day (manual capture)**

---

### Checkpoint Logic

* First 5 scans → Baseline
* Every 5 scans → New checkpoint

Checkpoints:

```
5 → baseline
10 → checkpoint 1
15 → checkpoint 2
...
```

---

### Alert Condition

* Significant drop vs previous checkpoint
  → Trigger **STRESS ALERT**

---

### Database Schema

```
Plant Table
- plant_id
- plant_name

Disease Table
- id
- plant_id (FK)
- image_path
- prediction
- confidence
- timestamp

VNIR Table
- id
- plant_id (FK)
- nir_green_ratio
- checkpoint_number
- timestamp
```

## 3.2 Mozhi — Chatbot + Memory

### Core Goals

* Conversational AI
* Context retention across sessions
* Local language support

---

### A. Chat System

* Main LLM: API-based (Llama 3.1 or similar)
* Fallback: Optional local LLM (Mac M4 Pro)

---

### B. Multilingual Support

* Translation Layer: **Bhashini (preferred)**

Pipeline:

```
User Input (Local Language)
→ Translate → English
→ LLM
→ Translate → Local Language
```

---

### C. Memory System (Revised Structure)

#### Levels

| Level   | Description        |
| ------- | ------------------ |
| Session | Current chat       |
| Daily   | Summary of 1 day   |
| Weekly  | 7 daily summaries  |
| Monthly | 4 weekly summaries |

---

### Memory Flow

```
Raw Chat
→ Daily Summary (cron)
→ Weekly Summary (cron)
→ Monthly Summary (cron)
```

---

### LLM Usage

| Task          | Model Type               |
| ------------- | ------------------------ |
| Chat          | Large LLM (API)          |
| Summarization | Small LLM (API or local) |

---

### Context Injection

Each new chat includes:

* Monthly summary
* Weekly summary
* Recent daily summaries
* Current session messages

## 3.3 Yukthi — RAG + Explainability

---

### A. RAG Pipeline

#### Components

* Embeddings: `bge-m3`
* Vector DB: ChromaDB (local)
* Sources:

  * KAU documents
  * ICAR guidelines

---

### Pipeline

```
Query
→ Embedding
→ Top-k Retrieval
→ Inject into LLM prompt
→ Response with citations
```

---

### Purpose

* Prevent hallucination
* Provide **verified agricultural advice**

---

### B. Grad-CAM Integration

* Library: `pytorch-grad-cam`
* Model: EfficientNet (.pth)

---

### Output

* Heatmap overlay on leaf
* Highlight:

  * Lesions
  * Damage areas
  * Color anomalies

---

### Role

* Grounds **Mizhi predictions**
* Improves **user trust**

---

## 3.4 Gathi — Backend System (Core Deliverable)

### Key Decision

👉 **No full mobile app required**

Primary deliverable:

* **FastAPI-based system**
* Optional lightweight frontend (Gradio / minimal UI)

---

### Backend (FastAPI)

Endpoints:

```
POST /diagnose
→ EfficientNet + GradCAM + rule-based response

POST /vnir
→ Thanal + ratio + checkpoint logic

POST /chat
→ Mozhi (memory + LLM + translation)

POST /rag
→ Yukthi retrieval
```

---

### Deployment Strategy

* Local system (primary)
* Optional:

  * Mac M4 Pro → LLM hosting (Ollama)
  * API fallback if local fails
