# NAVA — Module Description

This document provides detailed technical specifications for each module in the NAVA Phase 2 system. It serves as the authoritative reference for implementation decisions, module boundaries, data flows, and integration points.

---

## System Overview

NAVA Phase 2 is structured around four modules that transform the Phase 1 diagnostic pipeline into a comprehensive agricultural intelligence platform:

- **Mizhi** — Disease detection with explainable AI and VNIR-based early stress monitoring
- **Mozhi** — Multilingual conversational interface with contextual farm memory
- **Yukthi** — RAG-grounded advisory pipeline ensuring factual accuracy and safety
- **Gathi** — Backend orchestration and user interface layer

Each module is designed to be independently testable while maintaining clear integration contracts with other modules.

---

## Module 1 — Mizhi: Disease Detection & VNIR Monitoring

### Purpose
Mizhi combines two complementary diagnostic capabilities: reactive disease identification from visible symptoms (Phase 1 EfficientNet model) and proactive stress detection before symptoms appear (Thanal VNIR estimation). Both leverage the same user interaction — photographing a plant — but serve different detection timescales.

### Components

#### 1.1 EfficientNet-B0 Disease Classifier

**Model format:** PyTorch (`.pth` checkpoint)  
**Reason for PyTorch retention:** Grad-CAM integration via `pytorch-grad-cam` requires direct access to PyTorch model hooks and gradient computation. ONNX conversion would break XAI capability.

**Architecture:**
- Backbone: EfficientNet-B0 (compound-scaled CNN)
- Input: 224×224 RGB image, ImageNet normalization
- Output: 34-class softmax (7 crops × [healthy + diseases])
- Trained weights: Phase 1 checkpoint (`best_model.pth`, 94.54% validation accuracy)

**Inference pipeline:**
```python
# Pseudocode structure
image = load_and_preprocess(photo)  # resize, normalize
logits = model(image)
probs = softmax(logits)
predicted_class = argmax(probs)
confidence = max(probs)

if confidence >= 0.85:
    return (predicted_class, confidence, "RELIABLE")
else:
    return (predicted_class, confidence, "UNRELIABLE")
```

**Safety gate:** Confidence threshold ≥ 0.85 inherited from Phase 1. Predictions below this threshold are flagged but not suppressed — the UI shows both the prediction and the reliability warning.

**Dependencies:**
- PyTorch 2.x
- torchvision (for transforms)
- Pillow (image loading)

#### 1.2 Grad-CAM Explainable AI

**Library:** `pytorch-grad-cam`  
**Target layer:** Final convolutional layer of EfficientNet-B0 (`features[-1]` in torchvision implementation)

**Heatmap generation:**
```python
# Pseudocode structure
from pytorch_grad_cam import GradCAM
from pytorch_grad_cam.utils.image import show_cam_on_image

cam = GradCAM(model=efficientnet_model, target_layers=[model.features[-1]])
grayscale_cam = cam(input_tensor=image, targets=[predicted_class_index])
heatmap_overlay = show_cam_on_image(original_image, grayscale_cam, use_rgb=True)
```

**Output:** RGB image with red-yellow heatmap overlay highlighting the regions that most influenced the classification decision. This is returned alongside the disease label and confidence score.

**Integration point:** `/api/diagnose` endpoint returns a JSON payload containing:
```json
{
  "predicted_class": "Rice__Leaf_Blast",
  "confidence": 0.9234,
  "reliability": "RELIABLE",
  "gradcam_image_base64": "data:image/png;base64,iVBOR..."
}
```

#### 1.3 Thanal VNIR Engine

**Model format:** ONNX Runtime  
**Architecture:** UNet with Attention Gates (trained externally during competition)  
**Input:** 256×256 RGB image (leaf-only, segmented via HSV multi-cascade filtering)  
**Output:** Estimated NIR/Green spectral ratio (single float value)

**Preprocessing pipeline:**
1. Convert RGB to HSV colour space
2. Apply multi-cascade thresholding to isolate green leaf tissue
3. Crop to bounding box of largest connected component
4. Resize to 256×256
5. Run ONNX inference

**Checkpoint-based monitoring logic:**

Instead of absolute thresholds (which would require species-specific calibration data), Thanal uses a rolling baseline strategy:

```python
# Pseudocode
scans = get_scans_for_plant(plant_id)

if len(scans) < 5:
    status = f"Building baseline: {len(scans)}/5 scans collected"
    return {"status": status, "alert": False}

# Form checkpoints every 5 scans
checkpoints = []
for i in range(0, len(scans), 5):
    if i + 5 <= len(scans):
        chunk = scans[i:i+5]
        avg_ratio = mean([s.nir_green_ratio for s in chunk])
        checkpoints.append(avg_ratio)

# Compare last two checkpoints
if len(checkpoints) >= 2:
    delta = checkpoints[-2] - checkpoints[-1]
    if delta > ALERT_THRESHOLD:  # e.g., 0.15
        return {"status": "Stress detected", "alert": True, "delta": delta}
    else:
        return {"status": "Healthy trend", "alert": False, "delta": delta}
```

**Data storage:** SQLite with three tables:
- `plants` — user-created plant identifiers and metadata
- `vnir_scans` — raw scan history with timestamp, ratio, plant_id
- `vnir_checkpoints` — computed 5-scan averages with alert flags

**User interaction model:**

Two pathways feed the same VNIR pipeline:

1. **Direct monitoring:** User creates a named Plant ID in the UI. The system prompts (via scheduled notifications if mobile, or dashboard reminders if web) for photos at morning/afternoon/evening intervals. User taps "Scan now" → camera → Thanal processes → ratio logged.

2. **Disease detection cross-feed:** When user photographs a leaf for disease diagnosis, the UI prompts: "Add this scan to [Plant Name]'s VNIR history?" One-tap opt-in contributes the same photo to both pipelines.

**Dependencies:**
- ONNX Runtime (Python: `onnxruntime`, mobile: `onnxruntime_flutter` if mobile app is built)
- OpenCV (for HSV filtering and segmentation)
- NumPy

## Module 2 — Mozhi: Multilingual Chatbot & Contextual Memory

### Purpose
Mozhi transforms NAVA from a one-shot diagnostic tool into a persistent conversational agronomist. It remembers the farm's disease history, treatments applied, and outcomes observed across an entire growing season — without overwhelming the LLM context window. It also bridges the language gap by supporting Malayalam, Tamil, Hindi, and other Indian regional languages.

### Components

#### 2.1 Hierarchical Memory Architecture

**Problem:** Raw chat logs consume tokens linearly. A farmer using NAVA daily would exhaust Llama 3.1's 8K context window within a week.

**Solution:** Four-level memory compression pyramid.

**Level 1 — Live session context:**
- Current conversation's raw messages held in memory
- Cleared after session ends or after 24 hours of inactivity
- No persistent storage at this level

**Level 2 — Daily summary:**
- Cron job runs at midnight (APScheduler)
- Sends past 24 hours of raw messages to a lightweight LLM (e.g., Qwen2.5-1.5B via API or Ollama if locally hosted)
- Prompt: "Summarize this farmer's interactions today. Focus on: diseases identified, treatments recommended, questions asked, actions taken. Target: <200 tokens."
- Summary stored in SQLite `daily_summaries` table
- Raw messages cleared after summarization

**Level 3 — Weekly digest:**
- Cron job runs every Sunday at midnight
- Aggregates the past 7 daily summaries into a concise chronicle
- Prompt: "Condense these daily summaries into a single weekly digest. Target: <500 tokens."
- Stored in SQLite `weekly_digests` table

**Level 4 — Season chronicle:**
- Persistent record covering the full growing season
- Manually triggered or auto-generated monthly
- Contains: disease timeline, treatment history, crop stages, outcomes
- Injected into every LLM system prompt as farm context

**Context assembly (pre-LLM injection):**

When a new chat message arrives, the system prompt is assembled as:
```
You are an agricultural advisor for a farm in Kerala, India.

FARM HISTORY (read-only context):
[Season chronicle for this plant_id]
[Last 2 weeks' daily summaries]
[Current session's last 5 raw messages]

RETRIEVED KNOWLEDGE (from RAG — see Yukthi):
[Top 5 relevant document chunks]

USER MESSAGE:
[Current query]

Respond with practical advice. Cite sources when recommending treatments.
```

**SQLite schema (simplified):**
```sql
-- Raw session messages
CREATE TABLE chat_sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    plant_id TEXT,
    timestamp INTEGER,
    role TEXT,  -- 'user' | 'assistant'
    content TEXT
);

-- Daily summaries
CREATE TABLE daily_summaries (
    date TEXT,
    plant_id TEXT,
    summary TEXT,
    PRIMARY KEY (date, plant_id)
);

-- Weekly digests
CREATE TABLE weekly_digests (
    week TEXT,  -- e.g., '2025-W18'
    plant_id TEXT,
    digest TEXT,
    PRIMARY KEY (week, plant_id)
);

-- Season chronicle (long-term persistent)
CREATE TABLE season_chronicle (
    plant_id TEXT PRIMARY KEY,
    chronicle TEXT
);
```

**Cron job implementation:**
```python
# Pseudocode using APScheduler
from apscheduler.schedulers.background import BackgroundScheduler

scheduler = BackgroundScheduler()

# Daily summarization at midnight
scheduler.add_job(
    func=summarize_yesterday,
    trigger='cron',
    hour=0, minute=0
)

# Weekly digest every Sunday at midnight
scheduler.add_job(
    func=create_weekly_digest,
    trigger='cron',
    day_of_week='sun',
    hour=0, minute=0
)

scheduler.start()
```

**Dependencies:**
- APScheduler (cron job management)
- SQLite3 (built into Python, no separate install)
- LLM API for summarization (Hugging Face Inference API, OpenRouter, or Ollama if local)

#### 2.2 Multilingual Translation Bridge

**Problem:** Natively multilingual LLMs (Qwen, Sarvam, Llama-multilingual variants) have weak agricultural vocabulary in Indian regional languages. Disease names, pesticide compounds, and treatment protocols are often mistranslated or transliterated incorrectly.

**Solution:** Use Bhashini API as a translation layer around the English-language Llama 3.1 pipeline.

**Pipeline:**
```
Regional language input (Malayalam/Tamil/Hindi)
  ↓
Bhashini translate → English
  ↓
Llama 3.1 8B processes in English
  ↓
Bhashini translate ← English response
  ↓
Regional language output
```

**Entity preservation:**
Agricultural entities that must not be translated (pesticide brand names, chemical compounds, disease identifiers in scientific notation) are handled via:
1. Named Entity Recognition (NER) to tag entities pre-translation
2. Pass entities through translation unchanged
3. Re-insert entities into translated output

Example:
```
Input (Malayalam): "എന്റെ നെല്ലിന് ബ്ലാസ്റ്റ് രോഗം ഉണ്ട്, Tricyclazole എത്ര ഉപയോഗിക്കണം?"
  ↓ NER tags "Tricyclazole" as ENTITY
  ↓ Translate: "My rice has blast disease, how much [ENTITY:Tricyclazole] should I use?"
  ↓ Llama processes
  ↓ Response: "For blast control, apply [ENTITY:Tricyclazole] at 0.6g/L..."
  ↓ Translate back, entities unchanged
Output (Malayalam): "ബ്ലാസ്റ്റ് നിയന്ത്രണത്തിന്, Tricyclazole 0.6g/L എന്ന തോതിൽ..."
```

**Bhashini API integration:**
- Endpoint: `https://dhruva-api.bhashini.gov.in/services/inference/pipeline`
- Free tier: adequate for research/demo use
- Supported languages: Malayalam, Tamil, Hindi, Kannada, Telugu, Bengali, Gujarati, Marathi, Punjabi, Assamese, Odia
- Response time: ~1-2 seconds per translation

**Fallback:** If Bhashini API is unavailable or user language is unsupported, system defaults to English with a UI message: "Translation unavailable — responding in English."

**Dependencies:**
- `requests` (HTTP client for Bhashini API)
- `spacy` or `transformers` (optional, for NER entity tagging)

## Module 3 — Yukthi: RAG Pipeline & Explainable AI

### Purpose
Yukthi ensures that NAVA's treatment recommendations are factually accurate, traceable to authoritative sources, and transparently justified. It prevents the two most critical failure modes in agricultural AI: hallucinated chemical dosages (solved by RAG) and black-box predictions (solved by Grad-CAM).

### Components

#### 3.1 RAG (Retrieval-Augmented Generation) Pipeline

**Knowledge base sources:**

1. **Kerala Agricultural University (KAU) extension bulletins**
   - Freely available at kau.in/extension
   - Crop-specific disease management PDFs
   - Malayalam + English versions

2. **ICAR (Indian Council of Agricultural Research) guidelines**
   - Publicly available at icar.org.in
   - Integrated pest management protocols
   - Chemical safety datasheets

3. **State pesticide regulations**
   - Kerala State Department of Agriculture publications
   - Approved pesticide lists with dosage limits

**Document ingestion workflow:**

```python
# Pseudocode using LangChain + ChromaDB
from langchain_community.document_loaders import PyMuPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_community.vectorstores import Chroma

# 1. Load PDFs
docs = []
for pdf_path in knowledge_base_pdfs:
    loader = PyMuPDFLoader(pdf_path)
    docs.extend(loader.load())

# 2. Chunk with overlap to preserve context around chemical names
splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,        # ~350 tokens
    chunk_overlap=64,      # preserve context at boundaries
    separators=["\n\n", "\n", ". ", " "]
)
chunks = splitter.split_documents(docs)

# 3. Embed with multilingual model
embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-m3",
    model_kwargs={'device': 'cpu'}  # or 'cuda' if GPU available
)

# 4. Store in ChromaDB (persistent local vector store)
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chromadb_store"
)
```

**Retrieval at query time:**

```python
# Pseudocode
query = "How to treat rice blast?"
query_embedding = embeddings.embed_query(query)
results = vectorstore.similarity_search(query, k=5)  # top 5 chunks

# Extract text from results
context = "\n\n".join([doc.page_content for doc in results])

# Inject into LLM system prompt
system_prompt = f"""
You are an agricultural advisor. Answer based ONLY on the following verified sources.
If the sources don't contain the answer, say so.

SOURCES:
{context}

USER QUERY: {query}

Provide your answer and cite which source(s) you used.
"""
```

**LLM response with citations:**

The LLM is prompted to return responses in a structured format:

```
ANSWER:
For rice blast control, apply Tricyclazole at 0.6g/L as a foliar spray. 
Repeat after 10-15 days if symptoms persist.

SOURCES:
- KAU Rice Disease Management Bulletin (2023), page 14
- ICAR Integrated Pest Management for Rice, section 4.2
```

The `/api/rag` endpoint parses this and returns:
```json
{
  "answer": "For rice blast control, apply Tricyclazole at 0.6g/L...",
  "sources": [
    {"title": "KAU Rice Disease Management Bulletin (2023)", "page": 14},
    {"title": "ICAR Integrated Pest Management for Rice", "section": "4.2"}
  ]
}
```

**Safety mechanism:**

If the LLM generates a response that includes chemical names or dosages NOT present in the retrieved context chunks, the system flags it as potentially hallucinated and either:
1. Re-queries with a stricter prompt, or
2. Returns a fallback: "I don't have verified information on this treatment. Please consult an agronomist."

**Dependencies:**
- LangChain (RAG orchestration)
- ChromaDB (vector store — no server required, runs locally)
- `BAAI/bge-m3` via Hugging Face Transformers (multilingual embeddings)
- PyMuPDF or pdfplumber (PDF parsing)

**Initial setup time:** ~2-3 days
- 1 day: collect and organize PDFs from KAU/ICAR websites
- Half day: write ingestion script and populate ChromaDB
- 1 day: wire retrieval into `/api/rag` endpoint and test citation accuracy

#### 3.2 Grad-CAM Explainable AI

**Already covered under Mizhi (1.2)** — Yukthi is the conceptual owner of the XAI component, but implementation-wise it's tightly coupled to the PyTorch disease detection model, so the code lives in the Mizhi module. Yukthi's role is to ensure Grad-CAM output is consistently returned and displayed for every diagnosis.

**Cross-module contract:**
- Mizhi's `/api/diagnose` endpoint MUST return a `gradcam_image` field
- Gathi UI MUST display the heatmap overlay alongside the disease label
- If Grad-CAM generation fails (e.g., due to GPU OOM), return the diagnosis without heatmap but log the failure

## Module 4 — Gathi: Backend Orchestration & User Interface

### Purpose
Gathi is the integration layer that ties Mizhi, Mozhi, and Yukthi together. It exposes a unified API, manages request routing, handles the user interface, and provides offline fallback capabilities if time allows for mobile app development.

### Components

#### 4.1 FastAPI Backend

**Endpoints:**

```
POST /api/diagnose
  Input: { "image": "base64_encoded_image" }
  Process: 
    1. Decode image
    2. Run EfficientNet inference (Mizhi)
    3. Generate Grad-CAM overlay (Yukthi/Mizhi)
    4. Return diagnosis + confidence + heatmap
  Output: { "class": "...", "confidence": 0.92, "gradcam_image": "...", "reliability": "..." }

POST /api/vnir-upload
  Input: { "plant_id": "mango_01", "image": "base64_encoded_image" }
  Process:
    1. Run Thanal ONNX inference
    2. Log ratio to SQLite
    3. Check checkpoint logic
    4. Return current status + alert flag
  Output: { "ratio": 0.74, "status": "3/5 scans", "alert": false }

POST /api/chat
  Input: { "plant_id": "mango_01", "message": "How do I treat leaf spot?", "language": "en" }
  Process:
    1. If language != "en", translate via Bhashini (Mozhi)
    2. Retrieve farm history from SQLite (Mozhi)
    3. Query ChromaDB for relevant docs (Yukthi RAG)
    4. Assemble context + RAG chunks into system prompt
    5. Call LLM API (Hugging Face Inference / OpenRouter / Ollama)
    6. Translate response back if needed
    7. Store message in chat history
  Output: { "response": "...", "sources": [...] }

GET /api/plant/{plant_id}/history
  Returns: VNIR scan timeline + checkpoints + chat summary for UI display
```

**LLM API strategy:**

Since local model hosting is not in scope, Gathi calls external LLM APIs:

- **Primary option:** Hugging Face Inference API (free tier: 1000 requests/day, sufficient for demo)
  - Model: `meta-llama/Llama-3.1-8B-Instruct`
  - Endpoint: `https://api-inference.huggingface.co/models/meta-llama/Llama-3.1-8B-Instruct`

- **Fallback option:** OpenRouter (pay-per-token, more reliable uptime)
  - Gives access to Llama 3.1, GPT-4, Claude, etc. via unified API
  - Cost: ~$0.50/day for moderate demo usage

- **Summarization LLM:** Smaller model via Hugging Face (e.g., `Qwen/Qwen2.5-1.5B-Instruct`) for daily cron summarization jobs — faster and cheaper than using Llama 3.1 8B for this task.

**Orchestration logic:**

```python
# Pseudocode for /api/chat
@app.post("/api/chat")
async def chat_endpoint(request: ChatRequest):
    # 1. Translate input if needed
    if request.language != "en":
        english_message = bhashini_translate(request.message, request.language, "en")
    else:
        english_message = request.message
    
    # 2. Retrieve farm memory
    history = get_farm_history(request.plant_id)  # SQLite query
    
    # 3. RAG retrieval
    rag_context = chromadb_retrieve(english_message, k=5)
    
    # 4. Assemble system prompt
    system_prompt = build_prompt(history, rag_context)
    
    # 5. Call LLM API
    llm_response = call_huggingface_api(system_prompt, english_message)
    
    # 6. Translate back
    if request.language != "en":
        final_response = bhashini_translate(llm_response, "en", request.language)
    else:
        final_response = llm_response
    
    # 7. Store in chat history
    save_to_chat_history(request.plant_id, request.message, final_response)
    
    return {"response": final_response, "sources": extract_sources(llm_response)}
```

**Dependencies:**
- FastAPI (async Python web framework)
- `requests` (for calling external APIs)
- `python-multipart` (for handling file uploads)

## Future Work (Explicitly Out of Scope for This Phase)

The following features are documented as meaningful extensions but are not planned for the current 1-month development cycle:

### 1. IoT Sensor Fusion
**Concept:** Agnostic `/api/iot-ingest` endpoint that accepts real-time environmental readings (humidity, temperature, soil moisture, leaf wetness) from field sensor nodes (ESP32, weather APIs, etc.) and injects this context into the LLM system prompt — making disease risk assessment dynamically responsive to current field conditions.

**Why future work:** Requires hardware setup, sensor node programming, and integration testing with multiple IoT protocols. Scope is too large for 1-month timeline.

### 2. On-device LLM for Offline Advisory
**Concept:** Bundle a quantized small language model (e.g., Llama 3.2 1B-4bit, Phi-3-mini) in the mobile app for fully offline chat capability, removing dependency on internet connectivity for basic conversational advice.

**Why future work:** Model quantization + mobile optimization + testing is a 2-week effort. Current priority is getting the online pipeline working correctly.

### 3. Multi-label Disease Detection
**Concept:** Upgrade EfficientNet to detect multiple concurrent diseases per image (e.g., rice blast + bacterial blight simultaneously) by replacing the softmax output head with sigmoid + BCEWithLogitsLoss.

**Why future work:** Requires annotating the dataset with multi-label ground truth, which is time-intensive. Phase 1's single-label model is already strong; multi-label is an enhancement, not a necessity.

### 4. Mobile App (Flutter)
**Conditional:** If core functionality (Mizhi + Mozhi + Yukthi + Gradio UI) is complete by Week 3, mobile app development can begin. Otherwise, this moves to Future Work and the demo uses the web UI.
